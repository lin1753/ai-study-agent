# AI Study Agent - 智能体架构演进方案 (v2.0)

## 1. 架构目标
当前的系统基于“检索增强生成（RAG）+ LLM”的传统线性链路。为了解决幻觉严重、功能单一、无法自主纠错等问题，我们计划将系统全面升级为**基于 ReAct (Reasoning and Acting) 思想的真实智能体 (Agent) 架构**。

## 2. 核心架构设计

新系统将基于以下工作流：
`用户指令 -> Agent Controller (规划与拆解) -> 循环 [ 思考(Thought) -> 行动(Action/Tool) -> 观察(Observation) ] -> 整理回答 -> 记忆持久化`

### 2.1 核心模块拆解
1. **Agent Controller (大脑/中枢)**
   - 负责意图识别与任务拆解。
   - 维护短期的 ReAct 循环，防止系统陷入死循环（最高迭代限制）。
2. **Tool Registry (工具箱)**
   - **`rag_search_tool`**: 专职从本地数据库检索知识点/原题。隔离考试配置的干扰。
   - **`ocr_vision_tool`**: 通过嵌入 RapidOCR (ONNX Runtime) 实现纯粹的无依赖高性能本地 OCR 解析，用来兜底提取纯图片格式的 PPT 和 PDF，根除原来 Tesseract 存在的跨平台兼容性死角。
   - **`web_search_tool`**: （可选引申）当本地内容不足时，调用外部搜索进行补充。
   - **`exam_generator_tool`**: 根据考试权重配置（例如选/填/计比例），在获取局部知识点后进行定制化出题，负责彻底切割“解析”与“出题”逻辑，解决幻觉。
3. **Memory System (记忆组件)**
   - **短期记忆 (Session Context)**: 位于当前聊天的历史记录中。
   - **长期记忆 (User Profile)**: 构建学情画像。包含：用户常错题型总结、用户的表达偏好、用户薄弱科目设定等。
4. **Tone Modifier (语气过滤器)**
   - 新增输出后处理管道。对带有强烈“AI味”（“首先、其次、综上所述、作为AI模型”）的文本进行降级改写，或通过极强的 System Prompt 配合 Few-Shot 给定导师拟人风格。

## 3. 分阶段实施路线 (Roadmap)

### ✅ Phase 1: 底层基础设施改造与性能调优 (已完成)
- **成就**:
  1. 引入并手写了基于 Python 函数映射的轻量级 Agent Controller 工具链。
  2. 完成了 `DocumentParserTool` 与 `ExamGenerator` 的解耦，通过 Map-Reduce 大纲归并算法，根除超长 PDF 生成卡死与 Token 溢出问题。
  3. 将重型处理任务剥离至 Redis + RQ Background Worker 中，主线程解耦，支持超大并发上传。

### 🔄 Phase 1.5: 项目重构与模块优化 (当前阶段)
- **目标**: 解决 `backend` 目录下过于臃肿（意大利面条代码）、缺乏统一规范的问题，优化项目结构。
- **任务**:
  1. 标准化分层架构（将其划分为 `api`, `core`, `services`, `models`, `workers` 五个核心组件域）。
  2. 提取散落的配置与环境变量（Database URL, API Keys）。
  3. 梳理清理废弃与冗余脚本。
  4. 重构底层 PDF 知识解析链路，抛弃由于环境不一引起的 Tesseract C++ 报错，使用单文件 `RapidOCR ONNX` 代替，并配置 PDF 文件“即阅即焚”架构，只入全文档词向量 `pgvector` 而不再保留静态实体文件。

### Phase 2: 导师人设与记忆系统介入 (规划中)
- **目标**: 消除 AI 刻板语气，建立用户学习画像。
- **任务**:
  1. 在 `backend/teaching_styles.py` 中扩充并重构 `System Prompt`。
  2. 在数据库模型中增加 `UserProfile` 和 `MemoryLogs`。
  3. 在 Agent 反馈最终结果前，通过中间件拦截，如果判定套话过多，执行一次重写(Rewrite)。

### ✅ Phase 3: 工具箱扩展与前端流式反馈完善 (已完成)
- **目标**: 支持更多模态，缓解由于多次思考带来的推理高延迟。
- **任务**:
  1. 接入 DuckDuckGo/Tavily 做轻量化外网资料拉取。(✅ 已实装)
  2. 前端 `MainThread.jsx` 和 `ChatInterface.jsx` 改造：不再是单纯响应文本，需要通过 Server-Sent Events (SSE) 实时的推送 Agent 的内心戏，并与对应消息级（Message-Level）绑定（例如：页面显示 `[思考中...] 正在查找第一章例题...` -> `[工具调用] 本地没有找到原题，停止出题...`）。(✅ 已实装)
  3. **长文档解析实时状态同步**: 在文件上传或解析等待区间引入细粒度进度反馈（例如由后台 Worker 报告“正在抽取 PDF 文本...”、“正在调用大模型生成大纲...”），提供细粒度的前端 UI 进度提示展示，改善大文件处理导致的前端假死体验或纯轮询黑盒。(✅ 已实装)

### ✅ Phase 4: 解析引擎去正则化与鲁棒性加固 (已完成)
- **成就**:
  1. 通过移除 `llm_factory.py` 中存在缺陷的正则表达式转义修复机制，恢复了原生的 `json.loads` 解析方案，提升了文本分析准确率。
  2. 根除了由 LLM 返回的合法 JSON 字符串因过度替换反斜杠导致破坏后，频繁触发“解析失败备份”逻辑的顽疾，保证了前端渲染大纲数据的完美承接。
  3. 升级了前端状态管理模型，将追踪思考(`agentThoughts`)独立缓存方案改为与聊天列表项（`m.thoughts`）强绑定渲染方案，修复随消息丢失的 Agent Trace 日志显示 Bug。

### ✅ Phase 5: 主支线任务解耦与专属考核工具按需化 (已完成)
- **成就**:
  1. 移除了后台上传服务中耗时、强耦合的全幅自测题生成流程，将处理上传文档的主线（Main Thread）职责彻底纯粹化，仅保留切片和学习路径大纲（Roadmap/Plan）提取。大幅提升前端文件上传与解析响应速度，根除大纲数据的语义混杂污染。
  2. 摒弃支线对话（Branch Thread）简易且僵化的普通问答逻辑，为其全面引入了与主线看齐的 `StudyAgent` 机制。
  3. 将以往硬编码的试卷生成能力封装为了独立的 `ExamGenerator` 工具绑定于支线 Agent 中。在用户具体探讨某个知识点时，可按需动态生成测验题（On-Demand Tool Use）。
  4. 优化前端上传占位反馈显示，将全局毛玻璃防脱出遮罩转为沉浸于主内容区的日志进展面板，赋予用户流畅切分支的心流体验。

### ✅ Phase 6: 智能多模态解析与原题强化感知 (最新完成)
- **成就**:
  1. **OCR 回落机制接管扫描件**: 在 PDF/PPT 解析模块 (`utils/parsing.py`) 中嵌入基于 OnnxRuntime 的本地 `RapidOCR`，当标准提取文本过少（低密度/扫描件）时，自动触发视觉解析方案提取底层图文内容，彻底跨越无法提取扫描件知识点的瓶颈。
  2. **知识点与原题解构强耦合**: 大幅强化大模型 Roadmap 规划器指令（`llm_factory.py`），对发现的课后习题、思考题及例题不仅记录其内容，还通过 `original_questions` 字段精准挂载至 JSON 大纲下对于的知识点节点，使考点复原更深入。
  3. **支线 RAG 无痕弹药补充侧载（侧载模式）**: 新增专属增量补录 Worker 与 `/upload_supplementary` API 节点。前端允许用户在任一支线对话流中直接点击上传随手截图（jpg/png）或碎碎念资料，资料将仅仅经过 OCR/Embedding 的静默推送到本空间专有向量库（KnowledgeBlocks）中，绝不会破坏耗时重建原大纲，打造即拍即答的沉浸体验。

### ⏳ Phase 6.5: 支线加载心流过渡与防假死反馈 (规划中)
- **背景与痛点**: 用户在主线知识卡片中点击开启支线时，由于后端需要构建新线程上下文并初始化大模型，存在明显延迟。前端因死等异步请求导致界面仿佛毫无反应，等待数秒后才突然弹出侧边栏，体验僵硬。
- **目标**: 运用 Optimistic UI 提升前端瞬时响应感，为耗时的后台创建过程提供优雅的视觉过渡缓冲。
- **任务**:
  1. **响应解耦**: 点击支线按钮瞬间，剥离 `await API.createBranch` 的阻塞，先强制展开右侧 Branch Panel。
  2. **流程式引导占位**: 在右侧面板中注入加载骨架和分步提示动画（依次展示“初始化专属私教...” -> “挂载知识点...” 等文案），安抚等待焦虑。
  3. **无缝挂载**: 服务端返回 `thread_id` 后，通过状态驱动组件切换，平滑淡入 `ChatInterface`。

## 4. 当前技术栈面临的挑战点 & 对策
* **模型函数调用能力不足(Function Calling)**
  * *问题*: DeepSeek-r1:7b 在多轮 ReAct 下，直接输出符合 JSON Schema 规范的函数调用字符串可能会崩。
  * *对策*: 我们将采用 **XML/Markdown Tag 格式** 来替代死板的 JSON，小模型对 Tag (`<action>search</action>`) 的解析准确率远高于 JSON。
* **延迟 (Latency)**
  * *问题*: 分布式长考（多轮Prompt交互）会让首字输出的时间从 3 秒拉长到 10 秒以上。
  * *对策*: 强依赖前端 UI/UX 的补偿动画。在终端推送详细的 Agent Trace Log，给前端透传 State。
